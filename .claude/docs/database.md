# Database Architecture — Household Finance Planner
> Auto-generated by reading the codebase. Last updated: 2026-04-25. Audited and issues resolved: 2026-04-25 (v2.1).

## Overview

Supabase stores only household metadata, membership records, invite tokens, user public profiles, and the shared finance data blob. All of this is optional — the app works fully offline with localStorage as the primary store. Every function in `cloudInvites.ts` and `cloudFinance.ts` starts with `if (!supabaseConfigured) return` so they silently no-op when the env vars are absent. localStorage is always the source of truth for reads; Supabase is a sync layer that lets household members share data across devices and enables cross-device invite flows. Auth is handled entirely locally (SHA-256 password hashing or Google GIS JWT decode) — Supabase Auth is never used.

## RLS Policies

All tables use the same permissive pattern because auth is local (not Supabase Auth), so JWT-based RLS would block all anon-key requests. Security is enforced at the application layer instead.

| Table                   | Policy name  | For     | Using        | With check   |
|-------------------------|--------------|---------|--------------|--------------|
| households              | "allow_all"  | ALL     | (true)       | (true)       |
| household_memberships   | "allow_all"  | ALL     | (true)       | (true)       |
| invitations             | "allow_all"  | ALL     | (true)       | (true)       |
| household_invites       | "allow_all"  | ALL     | (true)       | (true)       |
| household_finance       | "allow_all"  | ALL     | (true)       | (true)       |
| user_profiles           | "allow_all"  | ALL     | (true)       | (true)       |

RLS is **enabled** on every table — the "allow_all" policy simply permits anon access, which is safe because the anon key is read-only from the client and the real secret (service role key) is never exposed.

## Data Flows

### A. User Sign Up / Sign In

**Email sign-up (`signUpEmail` in `localAuth.ts`):**
1. Check `hf-users` in localStorage for email collision.
2. Generate `userId` with `generateId()` (8-char base-36 string via `Math.random()`).
3. Hash password with `crypto.subtle.digest('SHA-256', ...)`.
4. Create `LocalUser` and `Household` objects in localStorage (`hf-users`, `hf-households`).
5. Write session to `hf-session` (`{ userId, householdId }`).
6. In `afterAuth()` (AuthContext): call `syncHousehold()` → upsert to Supabase `households`.
7. Call `syncMembership()` → upsert to `household_memberships` with role='owner'.
8. Call `syncUserProfile()` → upsert to `user_profiles`.
9. Call `refreshMembersFromCloud()` → query `household_memberships` + `user_profiles` and update local state.
10. Check `hf-pending-inv-token` / `hf-pending-invite` — if present, run accept flow (see flow E).
11. User lands on the main app shell.

**Email sign-in (`signInEmail` in `localAuth.ts`):**
Same as sign-up steps 6–11, but steps 1–5 are replaced by: find user in `hf-users`, verify SHA-256 hash.

---

### B. Household Creation

A household is created automatically at sign-up (or Google sign-in) as part of `createHousehold()` in `localAuth.ts`:
1. Generate `householdId` with `generateId()`.
2. Write `Household` to `hf-households` with `memberships: [{ userId, role: 'owner', joinedAt }]`.
3. Set `user.householdId` and write user to `hf-users`.
4. In `afterAuth()`: call `syncHousehold(id, name, createdBy)` → Supabase `households` upsert.
5. Call `syncMembership(householdId, userId, 'owner')` → Supabase `household_memberships` upsert.

Finance data is NOT seeded at creation — `household_finance` gets its first row when:
- The owner makes their first write (FinanceContext debounced push), OR
- FinanceContext detects local data exists but cloud has no row (seeds immediately on mount — "bootstrap seed" logic).

---

### C. Member Invitation — Email

1. Owner opens HouseholdSettings, types an email, clicks Send.
2. `createInvite('email', email)` is called in AuthContext → `createHouseholdInvite()` in `cloudInvites.ts`.
3. `generateInviteToken()` creates 32 random bytes → 64 hex chars (via `crypto.getRandomValues`).
4. `hashInviteToken(token)` → SHA-256 hash (via `crypto.subtle.digest`). Only the hash goes to DB.
5. Expire any previous pending invite for the same email in `household_invites` (UPDATE status='expired').
6. INSERT new row into `household_invites` with `method='email'`, `invited_email`, `token_hash`, `expires_at` = now + 7 days.
7. Return `CreatedHouseholdInvite` containing the raw `token` (in memory only — not logged, not stored).
8. Build invite URL: `https://household-finance-planner.com?inv={token}`.
9. POST to `/api/send-invite` (Vercel serverless) with `{ email, inviteUrl, householdName, inviterName }`.
10. Vercel function uses `RESEND_API_KEY` to deliver email via Resend.
11. If email fails (or RESEND_API_KEY not set), fall back to showing the URL in the UI for manual sharing.

---

### D. Member Invitation — Link

1. Owner clicks "Generate Link" in HouseholdSettings → Link tab.
2. `createInvite('link')` → `createHouseholdInvite(householdId, userId, 'link')`.
3. Revoke any existing pending link invite for the household (UPDATE status='revoked').
4. Same token generation + hashing as email flow (steps 3–6 above), but `method='link'`, `invited_email=null`.
5. Raw token stored in React state (`liveToken`) — only visible during the current browser session.
6. If the page is refreshed, the raw token is gone. The invite row still exists but the URL cannot be reconstructed without regenerating (which revokes the old row and creates a new one).
7. Owner copies the URL manually and shares via any channel.

---

### E. Accepting an Invite

**URL arrival:**
1. User visits `https://household-finance-planner.com?inv={token}`.
2. `main.tsx` detects `?inv=` param, writes raw token to `hf-pending-inv-token`, removes param from URL (no referrer leak).
3. If `?invite=` (legacy): writes invite ID to `hf-pending-invite`.

**After authentication (in `afterAuth()` in AuthContext):**
4. Read `hf-pending-inv-token` from localStorage.
5. Remove it immediately (so it isn't used twice).
6. Call `acceptHouseholdInvite(token, userId)` in `cloudInvites.ts`:
   a. Hash the token → `tokenHash`.
   b. Query `household_invites` WHERE `token_hash = tokenHash` (unique index lookup).
   c. Validate status (not revoked/expired/accepted) and expiry.
   d. If `method='email'`: UPDATE status='accepted' (one-time). If `method='link'`: leave 'pending' (multi-use).
   e. Call `syncMembership(householdId, userId, 'member')` → upsert to `household_memberships`.
   f. Call `getCloudHousehold(householdId)` → fetch household name from `households`.
   g. Return `{ householdId, householdName }`.
7. Call `applyHouseholdJoin(authedUser, householdId, householdName)`:
   a. Call `fetchHouseholdMemberProfiles(householdId)` → query `household_memberships` + `user_profiles`.
   b. Store each member's `LocalUser` in `hf-users` locally.
   c. Build a `Household` object from cloud member list and store in `hf-households`.
   d. Update `hf-session` with the shared `householdId`.
   e. Set `justJoined = true` → shows "Welcome to household" banner in the UI.
8. FinanceContext mounts with `key={household.id}` → fetches cloud finance data for the shared household → new member sees owner's data immediately.

---

### F. Finance Data — Adding Income/Expenses

1. User edits data in any tab (Income, Expenses, Savings, Goals).
2. Component calls a FinanceContext method (`addExpense`, `updateMember`, etc.).
3. All methods call `setData(updater)`:
   a. Apply the updater function to get `next` state.
   b. Write `next` to `localStorage` key `hf-data-{householdId}` immediately (synchronous).
   c. Set `hasLocalEditRef.current = true` (guard against cloud overwrite).
   d. Cancel any pending debounce timer, schedule new one for 1500 ms.
4. After 1500 ms of inactivity: `pushCloudFinanceData(householdId, next)` → Supabase `household_finance` upsert.
5. No optimistic rollback — if push fails silently (network error), local data is always the safe copy.

---

### G. Finance Data — Load on Mount

`FinanceProvider` mounts with `key={household.id}` (from App.tsx) — remounts fresh on household switch.

1. Load local data synchronously: `localStorage.getItem('hf-data-{householdId}')`.
2. Set `isLoading = true` only if Supabase is configured AND local data is empty (new member scenario).
3. Kick off `fetchCloudFinanceData(householdId)`:
   - SELECT `data` FROM `household_finance` WHERE `household_id = householdId` (single row lookup).
4. When cloud fetch resolves:
   - **Case A (cloud has data, user hasn't edited):** merge cloud into local using `mergeFinanceData(cloudData, localData)`. Cloud wins on all financial fields; local keeps `darkMode` + `language`. Save merged to localStorage.
   - **Case B (cloud has no row, local has data):** owner bootstraps cloud — call `pushCloudFinanceData` immediately (no debounce).
   - **Case C (both empty):** nothing to do.
5. Set `isLoading = false`. If loading was true, UI transitions from spinner to data.

---

### H. Members Tab — Loading Member List

Members are loaded at two points:

**On boot / after login (AuthContext `useEffect`):**
1. Restore session from `hf-session`.
2. Load user and household from localStorage.
3. Call `refreshMembersFromCloud(householdId, household)`:
   a. `fetchHouseholdMembers(householdId)`:
      - Query `household_memberships` WHERE `household_id = householdId` → get `[{ user_id, role, joined_at }]`.
      - Extract `userIds` array.
      - Query `user_profiles` WHERE `id IN (userIds)` → get `[{ id, name, email, avatar }]`.
      - Join in JS: build `CloudHouseholdMember[]` array.
   b. For each cloud member: call `upsertUserPublic()` to cache `LocalUser` in `hf-users`.
   c. Rebuild `Household.memberships` from cloud roles.
   d. Update local `hf-households` and React state.

**Members component (`Members.tsx`):**
- Reads `getMembers()` from AuthContext, which reads `household.memberships` and resolves each `userId` via `getUserById()` (localStorage lookup).
- No Supabase query at render time — data is already in local state from the boot refresh.

**HouseholdSettings** also has a manual "Refresh" button that calls `refreshInvites()` to re-fetch both pending invite lists from Supabase.

## Triggers and DB Functions

### `get_household_members_with_profiles(p_household_id text)`

Added in v2.1 to replace the two-query N+1 pattern in `fetchHouseholdMembers()`.

```sql
create or replace function public.get_household_members_with_profiles(p_household_id text)
returns table (user_id text, role text, joined_at timestamptz, name text, email text, avatar text)
language sql stable as $$
  select hm.user_id, hm.role, hm.joined_at,
    coalesce(up.name, '') as name,
    coalesce(up.email, '') as email,
    up.avatar
  from public.household_memberships hm
  left join public.user_profiles up on up.id = hm.user_id
  where hm.household_id = p_household_id
$$;
grant execute on function public.get_household_members_with_profiles(text) to anon;
```

Called via `supabase.rpc('get_household_members_with_profiles', { p_household_id })` in `cloudInvites.ts → fetchHouseholdMembers()`.

## Issues Found

### Security

1. ~~**`generateId()` uses `Math.random()` for entity IDs`**~~ ✅ **FIXED (v2.1)**
   - Was: `Math.random().toString(36).slice(2, 10)` — predictable, non-cryptographic 8-char IDs.
   - Fix applied: `generateId()` now calls `crypto.randomUUID()` in `src/lib/utils.ts`.
   - All new IDs (users, households, invitations, finance items) are now UUIDs.

2. **`allow_all` RLS with no row-level scoping**
   - File: `supabase/migration.sql` lines 47–49 and all subsequent allow_all blocks
   - What: Every table has `FOR ALL USING (true)` — anyone with the anon key can read, write, or delete any row in any table.
   - Why it matters: The anon key is embedded in the browser bundle (it's a VITE_ env var). Any user can open DevTools → Network and call Supabase REST directly to read all households' finance data.
   - Fix (accepted trade-off): Fully documented in the codebase — this is a known architectural choice because Supabase Auth is not used. The primary mitigation is that `householdId` must be known first. The real fix would be to add Supabase Auth or move financial data behind a server-side API.

3. **JWT not verified — `decodeGoogleJWT` trusts GIS output without signature check**
   - File: `src/lib/googleAuth.ts` lines 42–49
   - What: The JWT is decoded by base64-splitting the payload, no signature verification.
   - Why it matters: If an attacker can craft a JWT payload and inject it via a phishing page or XSS, they could sign in as any Google account. However, GIS tokens are delivered directly by Google's script in a trusted origin context, so practical risk is low with CSP headers in place.
   - Fix: Low priority given GIS delivers tokens directly; document the trust assumption clearly (already done in the comment on line 32).

8. ~~**Write functions discard `{ error }` from Supabase**~~ ✅ **FIXED (v2.1)**

### Data Integrity

4. **`user_profiles` has no FK to `household_memberships`**
   - File: `supabase/migration.sql` lines 104–110
   - What: A row can exist in `user_profiles` without any corresponding membership, and vice versa.
   - Why it matters: `fetchHouseholdMembers()` joins these in JS (not in SQL), meaning if `user_profiles` is missing for a member, they are silently dropped from the list (the `if (!profile) continue` on line 437 of `cloudInvites.ts`).
   - Fix: After any `syncMembership()` call, immediately call `syncUserProfile()` — this is already done in `afterAuth()` but not when accepting an invite on behalf of another user. The join in `fetchHouseholdMembers` should fall back to `household_memberships` data if profile is absent, or log a warning.

5. ~~**`hf-session` has no expiry — sessions persist forever**~~ ✅ **FIXED (v2.1)**
   - `AppSession` now has `expiresAt?: string` (ISO timestamp).
   - `persistSession()` sets a 30-day TTL. `getSession()` calls `clearSession()` and returns null if expired.

### N+1 / Performance

6. **`fetchHouseholdMembers()` fires two sequential Supabase queries**
   - File: `src/lib/cloudInvites.ts` lines 411–450
   - What: Query 1 → `household_memberships` to get `userIds`. Query 2 → `user_profiles` WHERE `id IN (userIds)`. These run in sequence.
   - Why it matters: Two network round-trips on every boot and on every manual refresh. In Postgres this could be a single query with a JOIN.
   - Fix: Use a Supabase foreign table relationship or a raw SQL RPC: `SELECT hm.user_id, hm.role, hm.joined_at, up.name, up.email, up.avatar FROM household_memberships hm JOIN user_profiles up ON hm.user_id = up.id WHERE hm.household_id = $1`.

7. **`fetchHouseholdMemberProfiles()` also fires two sequential queries (same pattern)**
   - File: `src/lib/cloudInvites.ts` lines 368–391
   - Fix: Same as above — consolidate into one JOIN query.

### Missing Error Handling

8. **`syncHousehold()`, `syncMembership()`, `syncUserProfile()`, `renameCloudHousehold()`, `removeCloudMembership()`, `cancelCloudInvitation()`, `revokeHouseholdInvite()` all discard errors**
   - Files: `src/lib/cloudInvites.ts` (multiple locations)
   - What: These functions await a Supabase call but do not check the returned `{ error }` object. Silent failures mean the cloud state drifts from local state without the user knowing.
   - Why it matters: If a membership sync fails (e.g., Supabase quota exceeded), the local household shows a member who doesn't exist in the cloud — the invite may appear to "work" locally but other members won't see the newcomer.
   - Fix: Add `const { error } = await supabase.from(...); if (error) console.error(...)` at minimum. Ideally surface toast errors for user-facing operations.

9. ~~**`importData()` in FinanceContext has no validation**~~ ✅ **FIXED (v2.1)**
   - `importData()` now validates: must be a non-null object (not array), and all array fields (`members`, `expenses`, `accounts`, `goals`, `history`) must actually be arrays if present. Throws a descriptive error otherwise.

## Glossary

| Term | Definition |
|------|-----------|
| `household` | A shared budget group. Each user belongs to exactly one household at a time. |
| `LocalUser` | The local representation of an authenticated user (email or Google). Stored in `hf-users`. |
| `HouseholdMember` | A finance-domain concept — a person whose income is tracked. Stored in `FinanceData.members` in localStorage. Different from `LocalUser`. |
| `HouseholdMembership` | An auth-domain concept — maps a `LocalUser` to a `Household` with a role (owner/member). |
| `FinanceData` | The full JSON blob for a household: members, expenses, accounts, goals, history, preferences. |
| `supabaseConfigured` | Boolean flag (`src/lib/supabase.ts`) — true only when both `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` are set. |
| `generateId()` | Returns `crypto.randomUUID()` — a cryptographically secure UUID v4. Used for all entity IDs. |
| `token_hash` | SHA-256 of a raw invite token. Only this is stored in Supabase. |
| `allow_all` | RLS policy pattern — permits all anon-key operations. Security enforced at app layer. |
| `mergeFinanceData()` | Pure function: cloud wins on financial fields; local wins on `darkMode` + `language`. |
| `hasLocalEditRef` | React ref in FinanceContext that prevents a slow cloud fetch from overwriting a concurrent local edit. |
| `isLocalEmpty()` | Checks if `FinanceData` arrays are all empty — used to decide whether to show the loading spinner and whether to seed the cloud. |
| `migrateIfNeeded()` | Idempotent migration from legacy `hf-accounts` format to the `hf-users` + `hf-households` format. |
| `PENDING_INV_TOKEN_KEY` | `hf-pending-inv-token` — stores a raw invite token from the URL until auth completes. |
| `PENDING_INVITE_KEY` | `hf-pending-invite` — stores a legacy invite ID from the URL until auth completes. |
| `justJoined` | AuthContext flag — true immediately after accepting an invite. Triggers the welcome banner. |
