# Auth & Household — Data Model & API

## TypeScript Types

### `LocalUser`

Represents a registered user stored in `localStorage`.

```typescript
interface LocalUser {
  id: string                            // crypto.randomUUID()
  name: string                          // Display name
  email: string                         // Unique identifier across auth providers
  avatar?: string                       // Profile photo URL (Google) or undefined (email auth)
  authProvider: 'google' | 'email'      // How the account was created
  householdId: string                   // The household this user belongs to
  createdAt: string                     // ISO timestamp
  passwordHash?: string                 // SHA-256 hash — email accounts only; Google accounts omit this
}
```

**Security note:** `passwordHash` is a hex-encoded SHA-256 digest of the user's password. Plain text is never stored, logged, or transmitted.

### `Household`

```typescript
interface Household {
  id: string                            // crypto.randomUUID()
  name: string                          // e.g. "The Smiths"
  createdBy: string                     // LocalUser.id of the owner
  memberships: HouseholdMembership[]    // All members including the owner
  createdAt: string                     // ISO timestamp
}
```

### `HouseholdMembership`

```typescript
interface HouseholdMembership {
  userId: string                        // LocalUser.id
  role: 'owner' | 'member'
  joinedAt: string                      // ISO timestamp
}
```

---

### `AppSession`

The active session stored in `localStorage`.

```typescript
interface AppSession {
  userId: string                        // The signed-in LocalUser.id
  householdId: string                   // The user's household
  expiresAt?: string                    // ISO timestamp; sessions expire after 30 days
}
```

On `AuthProvider` mount, `expiresAt` is checked. If the current time is past `expiresAt`, the session is cleared and the user is signed out automatically.

---

### `HouseholdInvite`

Stored in Supabase (`household_invites` table). Represents a pending invitation.

```typescript
interface HouseholdInvite {
  id: string                                          // UUID, generated by Supabase
  household_id: string                                // The inviting household's ID
  invited_email: string | null                        // Set for email invites; null for link invites
  method: 'email' | 'link'                            // Invite type
  status: 'pending' | 'accepted' | 'expired' | 'revoked'
  expires_at: string                                  // ISO timestamp; 7 days from creation
  created_by: string                                  // LocalUser.id of the inviter
  created_at: string                                  // ISO timestamp
  token_hash: string                                  // SHA-256 hash of the invite token (raw token never stored)
}
```

**Security note:** The raw invite token is embedded in the invite URL and is never stored anywhere. Only the SHA-256 hash of the token is stored in Supabase. On acceptance, the submitted token is hashed and compared to `token_hash`.

## localStorage Keys

All auth-related data is stored in `localStorage` under these keys:

| Key | Type | Contents |
|-----|------|----------|
| `hf-users` | `LocalUser[]` JSON | All registered users on this device |
| `hf-households` | `Household[]` JSON | All households created or joined on this device |
| `hf-session` | `AppSession` JSON | The active session; absent if signed out |
| `hf-data-{householdId}` | `FinanceData` JSON | All financial data for the household |

**Rule:** No component may read or write these keys directly. All access must go through `useAuth()` (for auth state) or `useFinance()` (for finance data).

## Supabase Tables

Supabase stores the coordination layer — data that must be shared across devices and users.

| Table | Primary key | Key columns | Purpose |
|-------|-------------|-------------|---------|
| `households` | `id` (text) | `name`, `created_by`, `created_at` | Household registry; enables cross-device household lookup |
| `household_memberships` | `(household_id, user_id)` | `role`, `joined_at` | Who belongs to which household |
| `household_invites` | `id` (uuid) | `token_hash`, `status`, `expires_at`, `method`, `invited_email` | Pending and historical invitations |
| `user_profiles` | `id` (text) | `name`, `email`, `avatar`, `updated_at` | Public user profile for Members tab display |
| `household_finance` | `household_id` (text) | `data` (jsonb), `updated_at` | Shared financial data blob; entire `FinanceData` object |

### Supabase RPC: `get_household_members_with_profiles`

```sql
-- Returns all members of a household with their profile data
get_household_members_with_profiles(p_household_id text)
RETURNS TABLE (
  user_id      text,
  role         text,
  joined_at    timestamptz,
  name         text,
  email        text,
  avatar       text,
  updated_at   timestamptz
)
```

This function joins `household_memberships` with `user_profiles` in a single round trip, avoiding N+1 queries for the Members tab.

## Finance Sync Rules

When cloud data is merged with local data (`mergeFinanceData`), the following conflict resolution applies:

| Field | Winner | Reason |
|-------|--------|--------|
| `members` | Cloud | Membership changes happen on any device |
| `expenses` | Cloud | Expense changes can happen on any device |
| `accounts` | Cloud | Account changes can happen on any device |
| `goals` | Cloud | Goal changes can happen on any device |
| `history` | Cloud | Snapshot changes can happen on any device |
| `currency` | Cloud | Currency changes affect all members |
| `locale` | Cloud | Locale changes affect all members |
| `emergencyBufferMonths` | Cloud | Shared household setting |
| `darkMode` | Local | Per-device UI preference |
| `language` | Local | Per-device UI preference |

The rationale: financial fields are shared household decisions — if any member updates them on any device, that update should propagate everywhere. UI preferences (dark mode, language) are personal and device-specific — they should never be overwritten by another member's choices.

## Password Storage

```
User enters password "hunter2"
      │
      ▼
TextEncoder.encode("hunter2") → Uint8Array
      │
      ▼
crypto.subtle.digest('SHA-256', bytes) → ArrayBuffer
      │
      ▼
Hex encode → "f52fbd32b2b3b86ff88ef6c490628285f482af15ddcb29541f94bcf526a3f6c7"
      │
      ▼
Stored in LocalUser.passwordHash
```

- Web Crypto API (`crypto.subtle`) — built into all modern browsers; no external libraries
- SHA-256 is a one-way hash — it cannot be reversed to recover the original password
- The plain text password exists only in memory during the hashing operation and is never written to localStorage, sent over the network, or logged

## Session Expiry

```typescript
// On AuthProvider mount:
const session = JSON.parse(localStorage.getItem('hf-session') ?? 'null')
if (session?.expiresAt && new Date(session.expiresAt) < new Date()) {
  localStorage.removeItem('hf-session')
  // User is redirected to auth page
}
```

New sessions are created with `expiresAt = now + 30 days`.
